# 静态知识库 vs 认知操作系统

## 对比背景

传统知识库（Notion、Obsidian、RAG、LLM Wiki 第一版）以"知识萃取和存储"为核心，而场景驱动认知闭环（wiki1/）以"知识组装和执行"为核心。这是两种根本不同的范式。

## 核心差异

| 维度 | 静态知识库 | 认知操作系统 |
|------|-----------|-------------|
| 核心目标 | 存储和检索知识 | 组装和执行知识 |
| 组织方式 | 以知识为中心（按类型/主题分类） | 以问题为中心（按场景分类） |
| AI 角色 | 检索工具（找到相关知识） | 执行引擎（按规则组装方案） |
| 知识形态 | 描述性（"是什么"） | 规范性（"怎么用"） |
| 概念处理 | 概念与实体混杂 | 概念（动词）与实体（名词）分离 |
| 方法定义 | 叙述性文字 | IPO 四元组（输入→处理→输出→工具） |
| 复用机制 | 无（每次从零检索） | 原子-组合双层复用 |
| 生长机制 | 人工手动维护 | YAML 反向校验自动生长 |
| 类比 | 图书馆 | 工厂 |

## 架构对比

### 静态知识库架构

```
raw/          ← 原始资料
entities/     ← 实体（人事物）
concepts/     ← 概念（混杂方法与工具）
comparisons/  ← 对比分析
queries/      ← 查询结果
```

问题：概念和实体混杂，方法缺乏可执行规范，没有组装机制。

### 认知操作系统架构

```
场景层 ← "我要解决什么问题"（组装规则）
  ↓
概念层 ← "用什么方法"（原子+组合，IPO 建模）
  ↓
实体层 ← "用到什么工具"（父子层级，4 种关系）
```

贯穿机制：IPO 闭环 + YAML 反向校验

## 能力对比

| 能力 | 静态知识库 | 认知操作系统 |
|------|-----------|-------------|
| 知识检索 | ✓ | ✓ |
| 知识萃取 | ✓ | ✓ |
| 概念抽象 | ✓ | ✓ |
| 方法复用 | ✗ | ✓（原子-组合双层） |
| 场景组装 | ✗ | ✓（composition 字段） |
| 自动生长 | ✗ | ✓（YAML 反向校验） |
| 端到端方案生成 | ✗ | ✓（场景→概念→实体→输出） |

## 范式转变

### 从"理解驱动"到"组装驱动"

- **理解驱动**: 依赖大模型本身的智能来理解知识并生成答案。不稳定、不可控、不可迭代。
- **组装驱动**: 依赖结构化规则的精度来编排知识。更稳定、更可控、更可迭代。

### 从"资料库"到"模式库"

```
资料库 → 知识库 → 模式库
我能看 → 我能查 → AI 能调用、能组装、能自我迭代
```

## 适用场景

| 场景 | 静态知识库 | 认知操作系统 |
|------|-----------|-------------|
| 个人笔记管理 | ✓ 足够 | 过度设计 |
| 知识检索查询 | ✓ 足够 | 过度设计 |
| AI 辅助写作（需要风格/方法论一致） | ✗ 不够 | ✓ 最佳 |
| AI 辅助决策（需要结构化推理） | ✗ 不够 | ✓ 最佳 |
| 知识库自动生长和演化 | ✗ 不够 | ✓ 最佳 |
| 企业级知识管理（多场景复用） | ✗ 不够 | ✓ 最佳 |

## 演进路径

静态知识库可以通过以下步骤升级为认知操作系统：

1. 将概念拆分为原子概念（IPO 建模）和组合概念（decomposition 编排）
2. 将实体从概念中独立出来，建立父子层级和 4 种关系
3. 定义场景层，为每个场景编写 composition 组装规则
4. 建立 YAML 反向校验机制
5. 配套 AI 操作规程（profile.md）
