# E3 AI 工作台调研大纲

基于 [[e3-ai-workbench|E3 AI 工作台项目]] 需求背景和 [[baisheng-survey-outline-system|百胜调研大纲体系]] 生成的项目专属调研大纲。本项目为 AI 增强型项目，调研重点从"系统功能覆盖"转向"岗位痛点与效率提升"。

## 生成依据

| 输入 | 来源 |
|------|------|
| 需求背景 | 客户希望通过 AI 改善 E3OMS 前端 UE 交互，专岗专治降本提效 |
| 一期场景 | 预测看板 + 自动套餐 + 库存捡漏预售 + 预售审单（3.5 个场景） |
| 目标岗位 | 运营（痛点最显著）、客服、商品 |
| 使用端 | E3OMS 中台 + 飞书双端 |
| 标准方案库 | 百胜调研大纲体系（适配 AI 项目裁剪） |

---

## 一、项目背景与目标调研

### 1.1 项目定位与期望

| 维度 | 内容 |
|------|------|
| **调研目的** | 明确客户对 AI 改造的核心期望和成功标准 |
| **调研重点** | AI 工作台 vs 传统功能开发的定位差异、管理层期望的提效指标 |
| **影响提示** | 决定方案设计重心（效率优先 vs 功能优先）、效果度量基线 |

**待确认**：
- [ ] 客户对"提效"的量化目标是什么？（如：操作步骤减少 X%、处理时间缩短 Y%）
- [ ] 一期上线的时间节点和预算范围？
- [ ] AI 工作台与现有 E3OMS 的关系？（嵌入 vs 独立）

### 1.2 组织架构与岗位职责

| 维度 | 内容 |
|------|------|
| **调研目的** | 了解运营、客服、商品三大岗位的组织架构和职责分工 |
| **调研重点** | 各岗位人数、日常工作流程、系统使用频率、与 E3OMS 的交互方式 |
| **影响提示** | 决定专岗专治的优先级和 AI 能力分配 |

**待确认**：
- [ ] 运营部门组织架构（是否分渠道运营/活动运营/商品运营？）
- [ ] 客服团队规模和排班方式
- [ ] 商品部门与运营部门的协作关系

---

## 二、岗位痛点深度调研

### 2.1 运营岗位痛点

| 维度 | 内容 |
|------|------|
| **调研目的** | 深入理解运营岗位的重复性操作痛点（已知为最显著） |
| **调研重点** | 日常高频操作清单、每项操作耗时、系统交互步骤数、跨系统切换频率 |
| **影响提示** | 决定一期场景优先级和 AI 方案设计重点 |

**已知痛点**：
- 系统交互影响工作效率（慢）
- 库存捡漏计算需手工完成
- 预售方案制定耗时长
- 数据看板需手工制作

**待确认**：
- [ ] 运营每天在 E3OMS 中操作的具体功能列表及耗时
- [ ] 哪些操作是"无价值重复劳动"（客户原话）
- [ ] 运营在飞书中的日常工作流（是否已有飞书使用习惯）

### 2.2 客服岗位痛点

| 维度 | 内容 |
|------|------|
| **调研目的** | 了解客服在预售订单处理中的痛点 |
| **调研重点** | 预售订单处理全链路步骤、跨部门沟通成本、异常处理频率 |
| **影响提示** | 决定预售审单自动化的设计范围 |

**已知痛点**：
- 客服需提前人工导出预售订单
- 需与商品人员跨部门确认到货
- 多平台数据需二次校准

**待确认**：
- [ ] 客服每天处理的预售订单量级
- [ ] 预售订单处理的峰值时段
- [ ] 当前使用的沟通工具和方式

### 2.3 商品岗位痛点

| 维度 | 内容 |
|------|------|
| **调研目的** | 了解商品运营在建新品套餐中的痛点 |
| **调研重点** | 套餐创建流程、渠道差异、盒规管理方式、批量操作需求 |
| **影响提示** | 决定自动套餐创建的规则引擎设计 |

**已知痛点**：
- 建新品套餐压力大
- 多渠道多规格需逐一手工创建

**待确认**：
- [ ] 每月新建套餐数量级
- [ ] 套餐创建的完整操作步骤（截图/录屏）
- [ ] 盒规数据来源和维护方式

## 三、一期场景详细调研

### 3.1 全链路预测分析看板

| 维度 | 内容 |
|------|------|
| **调研目的** | 明确看板需要展示的数据维度和预测需求 |
| **调研重点** | 当前看板制作流程、数据来源、更新频率、关注指标 |
| **影响提示** | 决定看板的数据模型和 AI 预测能力范围 |

**待确认**：
- [ ] 看板需要覆盖哪些维度？（渠道/商品/时间/区域？）
- [ ] 需要哪些预测能力？（销量预测/库存预警/趋势分析？）
- [ ] 看板在飞书端的展示形式偏好？（消息卡片/定时推送/交互式？）

### 3.2 自动套餐创建&观测工作台

| 维度 | 内容 |
|------|------|
| **调研目的** | 确认套餐创建的完整规则和异常处理 |
| **调研重点** | 三类套餐（ZH/LG/DYCH）的触发条件、盒规数据来源、创建后验证流程 |
| **影响提示** | 决定自动套餐引擎的规则配置和观测面板设计 |

**已知规则**（来自 [[e3-auto-package-rules|E3 自动创建套餐规则]]）：
- ZH 整盒套餐：全渠道，按盒规自动创建
- LG 双件套餐：得物/共鸣得物渠道
- DYCH 套餐：抖音拆盒渠道，1个/2个/整盒

**待确认**：
- [ ] 盒规数据在 E3OMS 中的存储位置和字段名
- [ ] 套餐创建后是否需要人工审核？
- [ ] 观测工作台需要展示哪些指标？（创建数量/成功率/异常数？）

### 3.3 库存捡漏+预售方案

| 维度 | 内容 |
|------|------|
| **调研目的** | 确认各渠道捡漏规则和预售方案制定逻辑 |
| **调研重点** | 当天捡漏 vs 次日捡漏的触发条件、各渠道差异化规则、预售方案制定流程 |
| **影响提示** | 决定捡漏引擎和预售方案推荐算法的设计 |

**已知规则**（来自 [[e3-channel-inventory-pickup|渠道捡漏规则]]）：
- 微信小店：等退款释放后捡漏
- 抽盒机：按实时退款数量捡漏
- 天猫：分货数 - 实际销售数
- 次日捡漏：全部订单占库后统一捡漏

**待确认**：
- [ ] 捡漏操作的当前人工耗时
- [ ] 预售方案的制定逻辑和参考因素
- [ ] 捡漏结果需要在飞书中如何呈现？

### 3.4 预售审单自动化（场景 3.5）

| 维度 | 内容 |
|------|------|
| **调研目的** | 确认预售审单的完整规则和自动化边界 |
| **调研重点** | 数据汇总→筛选→占库存→审核全链路规则、异常处理、渠道差异 |
| **影响提示** | 决定预售审单自动化的流程设计和飞书通知策略 |

**已知规则**（来自 [[e3-presale-review-rules|预售审单规则]]）：
- 提前 7 天触发预售提醒
- 按用户下单时间顺序占库存
- 排除补发单/拆盒单/换货单/手工单/复制单
- 超过一个月的预售订单需激活

**待确认**：
- [ ] 预售订单的日均处理量
- [ ] 部分到货场景的发生频率
- [ ] 飞书端需要哪些审批交互？（一键审核/批量操作？）

## 四、技术环境调研

### 4.1 E3OMS 系统现状

| 维度 | 内容 |
|------|------|
| **调研目的** | 了解 E3OMS 当前版本、接口能力和扩展性 |
| **调研重点** | E3OMS 版本、API 开放程度、是否支持嵌入式 AI 组件 |
| **影响提示** | 决定 AI 工作台的技术集成方案 |

**待确认**：
- [ ] E3OMS 当前版本和架构
- [ ] 是否开放 API 供 AI 工作台调用？
- [ ] 前端是否支持 iframe/微前端嵌入？

### 4.2 飞书使用现状

| 维度 | 内容 |
|------|------|
| **调研目的** | 了解飞书在企业内部的使用深度和开放能力 |
| **调研重点** | 飞书版本、是否使用飞书机器人/消息卡片/审批流 |
| **影响提示** | 决定飞书端 AI 助手的交互形式 |

**待确认**：
- [ ] 飞书在企业内的使用覆盖率
- [ ] 是否已使用飞书审批、飞书文档等功能？
- [ ] 是否有飞书机器人开发经验？

### 4.3 AI 能力需求

| 维度 | 内容 |
|------|------|
| **调研目的** | 明确 AI 工作台需要的 AI 能力类型 |
| **调研重点** | 需要哪些 AI 能力？（规则引擎/预测模型/NLP/推荐算法） |
| **影响提示** | 决定 AI 技术选型和模型训练策略 |

## 调研执行计划

| 模块 | 预计时长 | 参与角色 | 优先级 |
|------|---------|---------|--------|
| 项目背景与目标 | 0.5 天 | 项目负责人、IT 负责人 | P0 |
| 运营岗位痛点 | 1 天 | 运营主管、运营专员 | P0 |
| 客服岗位痛点 | 0.5 天 | 客服主管 | P1 |
| 商品岗位痛点 | 0.5 天 | 商品主管 | P1 |
| 一期场景详细调研 | 1.5 天 | 运营+客服+商品+IT | P0 |
| 技术环境调研 | 0.5 天 | IT 架构师 | P1 |

**总预计**：4.5 天
