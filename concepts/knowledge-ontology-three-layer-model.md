# 知识本体三层模型

基于 LLM Wiki 的知识本体建模方法，构建 **场景 → 概念 → 实体** 三层结构，形成场景和问题驱动的认知闭环知识体系。

## 核心理念

> 构建知识库和知识图谱不是目的，应用知识库中的知识点、利用知识点的关联进行推理、形成知识点的动态组装最终解决场景下的问题才是关键。

三层模型将知识从"静态存储"升级为"动态组装"——任何场景的支撑都以底层概念和实体为基础，通过概念实体的组装编排完成场景和问题的解决。

## 三层结构

```
场景层 (Scenarios)
  ↓ 组装编排
概念层 (Concepts)
  ↓ 引用调用
实体层 (Entities)
```

### 场景层（Scenarios）

场景本质是**应用已有的概念和实体按步骤进行组装的过程**。

- 基于知识库历史文章抽取，归类为 5 大类 47 个场景
- 每个场景详细定义：阶段划分、每个阶段引用的概念或实体、知识点组装的步骤和流程
- 场景是问题驱动的入口——用户带着问题来，系统匹配场景，动态组装知识点

### 概念层（Concepts）

概念方法层的重点：**必须给出具体的输入、输出，包括工具技术的应用**。

- **原子概念**：不可再分的基础方法，包含 IPO 四元组（Input / Process.Steps / Output）
- **组合概念**：由多个子方法串联而成的复合方法，包含 decomposition（分解结构）
- 将核心的动态逻辑引入概念方法，解决传统流程分析的静态思路
- 支持概念间的分层：复合概念可拆分为子概念

### 实体层（Entities）

引入父实体和子实体的分层结构，体现实体间的层次关系。

实体分类：
- **工具/产品**：软件、平台、框架（如 entity-knowledge.yaml）
- **方法论实体**：问题解决方法、学习框架（如 entity-problem-solving.yaml）
- **人物/底层技术/哲学概念**（如 entity-other.yaml）

## 与传统本体建模的对比

| 维度 | 传统结构化本体 | 知识本体三层模型 |
|------|---------------|-----------------|
| 核心对象 | 数据库表 | 概念和实体 |
| 数据来源 | IT 系统 | 历史资料文档抽取 |
| 目标 | 数据管理 | 场景驱动的知识组装 |
| 动态性 | 静态 Schema | 动态进化 |
| 应用方式 | CRUD 查询 | 场景匹配 → 概念组装 → 实体调取 |

## 知识图谱可视化

基于三层模型可通过 AI 编程实现可视化：
- **分层网络图**：场景层（顶）→ 概念层（中）→ 实体层（底）
- **场景支撑图**：选中某场景，动态列出支撑该场景的概念和实体
- **泳道图**：分阶段展示场景执行过程
- **力导向图**：展示完整知识图谱网络

## 自我进化能力

知识库是动态的，具备自我进化能力：
- 每次写作完成后，AI 分析是否需要增加新场景或概念
- 检测已有概念实体是否存在死链需要修正
- 随新场景和问题的引入，知识库不断完善和进化
- 这与 [[llm-wiki|LLM Wiki（Karpathy 模式）]] 的核心理念一致

## 与 RAG 的对比

传统 RAG 知识库存在知识幻觉问题。基于 LLM Wiki 的场景驱动知识库：
- 知识点经过编译和交叉引用，矛盾已被标记
- 场景层提供结构化的组装路径，而非碎片化检索
- 能更好支撑方案类文档写作（立项报告、采购文件、招标文件、标准规范等）
