# 雅戈尔 E3+ 调研分析

基于 [[youngor-e3-survey-log|雅戈尔 E3+ 业务调研日志]] 的系统性分析，提炼调研执行模式和可复用的方法论。

## 调研执行模式

### 五段式调研结构

```
主数据及库存 → 全渠道业务 → 电商正向 → 电商逆向 → 财务对账
```

与 [[baisheng-survey-outline-system|标准调研大纲]] 的 6 大模块高度对应：

| 标准模块 | 雅戈尔调研 | 覆盖度 |
|---------|-----------|--------|
| IT 部门调研 | 未单独成篇（融入各模块） | 部分 |
| 基础主数据 | 主数据及库存 | ✅ |
| 商品主数据 | 主数据及库存 + 字段映射 | ✅ |
| 库存管理 | 主数据及库存 | ✅ |
| 电商业务 | 全渠道 + 正向 + 逆向 | ✅ |
| 电商对账 | 财务对账 | ✅ |

### 调研记录格式

每条调研记录遵循统一结构：

```
调研内容：具体调研问题（含子问题列表）
业务现状：客户当前的实际情况
涉及系统：关联的外部系统
提升方向：E3+ 升级后的改进方案
备注：待确认事项或风险点
```

## AS-IS 关键发现

### 系统架构现状

```
酷客多/喵街/天虹/杉杉奥莱 → 用友中台 → E3 OMS
淘宝/京东/唯品会/抖音/快手 → E3 OMS
线下门店 → POS → 中台 → E3
```

**核心问题**：部分渠道经用友中台中转，链路长、时效低。

### 库存管理现状

- 全量同步：每晚 21 点 POS → 中台 → 差异化更新 → E3
- 增量同步：门店库存变动实时推送
- 中台按比例分配库存到各渠道

### 订单履约现状

- 路由在中台完成，E3 不做自动合并/拆分
- 门店发货需经 E3→中台→POS 三步
- 退货退款流程涉及多个系统协同

## TO-BE 提升方向

| 领域 | 现状痛点 | 提升方案 |
|------|---------|---------|
| 主数据 | 依赖用友中台中转 | E3+ 直接对接，增加补偿机制 |
| 库存 | 多系统库存同步链路长 | 统一库存中心，单据对接替代全量同步 |
| 订单 | 路由在中台，E3 无自动化 | E3+ 自动化审核 + 灵活适配策略 |
| 逆向 | 退货手工操作，换货打标处理 | 自动下载退换货申请，包裹匹配 |
| 对账 | 账单格式不统一，手工导入 | 自动对账 + 差异处理 |

## 项目交付物分析

### 功能需求清单（1055 条）

| 模块 | 典型需求 |
|------|---------|
| 电商管理 | 门店接单/拒单统计、履约效率统计、自动重路由、拆单策略 |
| 分单策略 | 普通订单分单、分销订单分单、即时零售分单 |
| 订单管理 | 零售订单强制取消 |

### 接口清单（89 条）

| 外部系统 | 接口数 | 典型接口 |
|---------|--------|---------|
| 微商城 | 9+ | 商品列表、库存同步、订单推送、售后 |
| WMS | 5+ | 订单推送、出库回写、取消、唯品会发货 |
| POS | 3+ | 订单取消、发货回写 |
| 用友中台 | 多个 | 主数据同步 |

### 迭代计划（5 个 Sprint）

```
Sprint 1 (2周): 分单策略 + 主数据接口
Sprint 2 (2周): 分单策略 + WMS对接 + 唯品会
Sprint 3 (2周): 财务对账 + POS/微商城接口
Sprint 4 (2周): 收尾优化
Sprint 5 (2周): 发布
```

总工期约 10 周。
